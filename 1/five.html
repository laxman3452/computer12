Database Normalization (1NF, 2NF, 3NF)

Reduce Redundancy • Improve Data Integrity • Organize Data Efficiently

📌 What is Normalization?

Normalization is the process of organizing data in a database to:

  • Reduce data redundancy
  • Improve data integrity
  • Avoid anomalies (insert, update, delete problems)

⚠️ Data Redundancy (Example)

Data redundancy means repeating the same data multiple times.

Student IDNameCourseTeacher
1RamDBMSMr. Sharma
2SitaDBMSMr. Sharma
3HariDBMSMr. Sharma

👉 "Mr. Sharma" is repeated → this is redundancy.

🔐 Data Integrity (Example)

Data integrity means data is accurate and consistent.

If teacher name changes from "Mr. Sharma" → "Mr. Shrestha":

  • Without normalization → multiple rows must be updated ❌ (error risk)
  • With normalization → update once only ✅

1NF (First Normal Form)

  • No repeating groups
  • Each column must have atomic values
StudentPhone
Ram9801, 9812 ❌

✔ Fix (1NF):

StudentPhone
Ram9801
Ram9812

2NF (Second Normal Form)

  • Must be in 1NF
  • Remove partial dependency

Partial Dependency Example:

StudentIDCourseIDStudentNameCourseName
1C1RamDBMS

👉 StudentName depends only on StudentID, not full key (StudentID + CourseID)

3NF (Third Normal Form)

  • Must be in 2NF
  • Remove transitive dependency

Transitive Dependency Example:

StudentIDDeptIDDeptName
1D1Computer Science

👉 StudentID → DeptID → DeptName (transitive dependency)

📘 Partial Dependency (2NF Concept)

Definition: A partial dependency occurs when a non-key attribute depends on only part of a composite primary key.

Key Idea: Happens only when primary key is composite (multiple columns)

❌ Problem Table (Before 2NF)

StudentID CourseID StudentName CourseName Instructor
S1C1RamDBMSMr. Sharma
S1C2RamDSAMr. Verma
S2C1SitaDBMSMr. Sharma

❌ Problem: StudentName depends only on StudentID, not full key (StudentID + CourseID)

✔ Solution (2NF Tables)

Student
S1 - Ram
S2 - Sita
Course
C1 - DBMS - Mr. Sharma
C2 - DSA - Mr. Verma
Enrollment
S1 - C1
S1 - C2
S2 - C1

📗 Transitive Dependency (3NF Concept)

Definition: A transitive dependency occurs when a non-key attribute depends on another non-key attribute.

Example: StudentID → DeptID → DeptName

❌ Problem Table (Before 3NF)

StudentID Name DeptID DeptName DeptLocation
S1RamD1CSBlock A
S2SitaD1CSBlock A
S3HariD2ITBlock B

❌ Issue: DeptName depends on DeptID, not directly on StudentID

🔥 3NF Step-by-Step Flow (VERY IMPORTANT)

STEP 1: Start with 2NF table (Student + Dept info together)
STEP 2: Identify dependency chain:
StudentID → DeptID → DeptName
STEP 3: Detect transitive dependency
STEP 4: Split tables

✔ Final 3NF Tables

Student Table
S1 - Ram - D1
S2 - Sita - D1
S3 - Hari - D2
Department Table
D1 - CS - Block A
D2 - IT - Block B
🎯 Result: No redundancy, no indirect dependency, high data integrity

📌 Final Summary

  • 1NF: Remove repeating groups
  • 2NF: Remove partial dependency
  • 3NF: Remove transitive dependency
